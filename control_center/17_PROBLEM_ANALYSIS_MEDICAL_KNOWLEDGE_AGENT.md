## 题目一句话

开发一个学科知识整合智能体，把 7 本医学教材解析成知识图谱，跨教材识别重复、互补和缺失，在不超过原始体量 30% 的前提下输出精华整合版本，并提供有原文引用的 RAG 问答与教师反馈迭代能力。

## 老师视角的核心判断

老师不会只看“能不能聊天”，而会看：

- 多本教材是否被系统化拆成知识点和关系。
- 重复、互补、缺失是否有可解释的判定依据。
- 30% 压缩是否有教学完整性保障，而不是随机删内容。
- RAG 回答是否每条都有教材、章节、页码引用。
- Agent 架构文档是否能解释为什么这样设计，以及有什么取舍。

## 主要线索

1. 多源教材进入系统：PDF/Markdown/TXT 必做，批量上传、解析状态、章节结构展示。
2. 单本教材先成图：抽取知识点、定义、类别、章节、页码，并建立至少 3 类关系。
3. 跨教材合并去重：识别同义/近义知识点，输出 `merge / keep / remove` 决策和理由。
4. 压缩到 30% 以内：展示原始字数、整合后字数、压缩比，解释教学逻辑不断裂。
5. RAG 引用问答：基于教材 chunk 检索回答，必须附来源；找不到就说“当前知识库中未找到相关信息”。
6. 教师反馈迭代：真实教师不可用时，用模拟教师反馈实现保留、拆分、合并、删除至少一条决策。

## 关键细节定义

- 30% 压缩：`压缩比 = 整合后保留内容总字数 / 原始正文总字数 × 100%`，必须做到 `<= 30%`。例如原始正文 100000 字，整合后不能超过 30000 字。
- 原始正文总字数：应尽量排除目录、页眉页脚、重复空白；如果比赛中只能粗略统计，必须在报告中说明统计口径。
- 整合后字数：不能只统计知识点名称，应统计保留知识点定义、解释和必要摘要的总字数。
- 知识点粒度：一个节点可以是概念、方法、现象、机制、疾病、病理过程等，但必须在需求分析中定义清楚。
- 重复判定：同一医学概念的不同表述可以合并；上下位概念、相关概念、互补内容不能粗暴合并。
- 教学完整性：压缩后要说明前置依赖链路是否断裂，哪些内容被保留为关键路径，哪些内容被删除为重复讲解。
- RAG 引用：回答必须返回教材名、章节、页码、相关度和原文片段；找不到答案必须说未找到。

## 建议技术路线

- 前端：React + Vite 单页应用。
- 后端：FastAPI，负责解析、抽取、整合、RAG、反馈修改。
- 图谱：ECharts Graph 优先，保证点击详情、缩放拖拽、颜色区分来源、频次大小映射。
- 模型：使用已有 API 做知识点抽取、RAG 生成、合并理由解释。
- 存储：MVP 使用 JSON/SQLite 保存教材结构、节点、边、合并决策、教师反馈。
- 检索：优先 API embedding；若不稳，用 TF-IDF/关键词检索兜底，并在文档说明取舍。
- 数据策略：先用 2 本教材跑通闭环，再扩展到 7 本生成报告。

## Analysis 分支结论

- 标准全栈路线可行，但必须避免一开始硬做全量 7 本实时处理。
- 首屏和演示要围绕图谱与引用问答，因为这两个最容易让老师判断“像不像真的系统”。
- 文档不是附属品，`docs/Agent 架构说明.md` 是 20 分核心评分项，应与开发并行。
- 30% 压缩、知识点粒度、重复判定、引用来源、教师反馈修改、教学完整性说明都是容易被遗漏但会影响评分的细节。
