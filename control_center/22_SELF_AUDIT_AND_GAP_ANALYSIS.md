# 自我筛查与差距分析

时间：2026-05-10 11:36 CST

定位：Root 总控台对当前作品进行反向审查，目标是防止“界面看起来完整，但不符合赛题真实验收”的风险。

## 结论

没有完美达到。

当前项目不是方向错误：它确实围绕“医学教材知识整合智能体”建立了前端、后端、图谱、整合、压缩、RAG、反馈和文档路线，也已经具备本地可验证的 MVP 闭环。

但如果按赛题严格验收、公网体验和“Agent 真实能力”来要求，当前仍有明显差距。现在必须停止自我安慰，把剩余时间用于补齐提交闭环、如实写清能力边界，并优先修最影响评分的硬伤。

## 四种以上功能是否达标

从“本地后端 + 前端演示”的角度，功能数量已经超过四种：

1. 教材解析：后端支持 PDF、Markdown、TXT；`/api/documents` 支持路径和 multipart 上传。
2. 知识图谱：后端生成节点和边；前端用 ECharts 展示图谱。
3. 跨教材整合：后端输出 `merge / keep / remove / split` 类型决策、理由和置信度。
4. 30% 压缩审计：后端计算原始字数、整合后字数、压缩比和达标状态。
5. RAG 问答：后端 `/api/rag/query` 返回答案和教材、章节、页码、片段引用。
6. 教师反馈：后端 `/api/feedback` 可修改一条整合决策；前端有模拟反馈演示。
7. Agent 架构文档：`docs/Agent 架构说明.md` 已存在。

因此，“四种功能以上”在演示和代码层面是达标的。

但“完美达到”不成立，因为公网部署和真实交互没有完全覆盖这些能力。

## 最大硬伤

1. 公网链接当前主要是静态前端 Demo，后端没有作为公网可访问服务部署。
2. 前端没有真实文件上传区，未把 `/api/documents` 作为主要交互入口暴露给评委。
3. 前端 RAG 和反馈以演示状态为主，真实 `/api/rag/query`、`/api/feedback` 未成为公网闭环。
4. 后端 RAG 是关键词打分检索，不是完整 embedding、向量库、rerank 的 RAG。
5. 知识抽取和整合主要依赖规则、关键词、同义词表，不是对每章节调用 LLM 的强语义抽取。
6. 当前 smoke test 只跑 2 本教材，不是 7 本教材全量整合。
7. “缺失知识点”识别不足，当前更偏重复、保留、删除和局部互补。
8. 教师多轮对话是模拟反馈，不是真实自然语言多轮会话或教师在线迭代。
9. 压缩比能计算并满足小于 30%，但整合后内容可能偏摘要预算，不等于高质量完整教材压缩生成。
10. README 和报告里仍有过时占位或部署口径，需要立即回填 GitHub Pages、GitHub 仓库和限制说明。

## 不是根本框架错误的部分

当前框架仍然合理：

- React + FastAPI + ECharts 对应 Web 单页、API、图谱可视化的评分方向。
- 后端模块顺序与赛题一致：解析 -> 抽取 -> 图谱 -> 整合 -> 压缩 -> RAG -> 反馈。
- 文档结构覆盖需求分析、系统设计、Agent 架构说明、整合报告和技术报告草稿。
- 前端“教师四问”正好对齐评委关心的合并依据、来源追溯、压缩阈值和引用问答。

问题不是“搭错了楼”，而是“楼里有几层还只是样板间”。现在要用文档和最小补救把风险收束。
