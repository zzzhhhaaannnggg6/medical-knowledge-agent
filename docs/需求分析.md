# 需求分析

## 1. 背景与目标

医学教材体量大、章节多、概念关系密集。学生和教师在复习或备课时，需要知道不同教材之间哪些内容重复、哪些内容互补、哪些知识点是前置依赖。本项目目标是构建一个医学教材知识整合智能体，把多本教材整理为可视化知识图谱，并输出可解释的压缩整合结果。

一句话目标：把医学教材解析成知识图谱，在不超过原始体量 30% 的前提下生成精华整合版本，并提供带原文引用的问答与教师反馈迭代能力。

## 2. 用户角色

- 教师：关注整合依据、教学完整性、压缩后是否漏掉关键链路。
- 学生：关注能否快速理解重点、追溯原文、围绕问题检索答案。
- 评委：关注系统是否可运行、功能闭环是否完整、文档是否可复现。

## 3. 功能需求

### 3.1 教材上传与解析

- 支持 PDF、Markdown、TXT。
- 展示文件名、格式、大小、解析状态、章节数量。
- PDF 解析优先保证稳定，不追求一开始处理全量 824MB。
- 章节结构至少包含章节名、页码或页范围、字符数。

### 3.2 知识点抽取与图谱

- 从教材章节中抽取知识点名称、类型、定义或摘要。
- 保留来源教材、章节、页码。
- 生成节点和边，关系至少覆盖 `contains`、`prerequisite`、`related`。
- 前端图谱支持缩放、拖拽、点击节点查看详情。

### 3.3 跨教材整合

- 对 2 本以上教材生成跨教材整合决策。
- 每条决策包含动作、理由、置信度、受影响节点、结果节点。
- 决策动作至少包含 `merge`、`keep`、`remove`。
- 系统需要解释为什么合并、为什么保留、为什么删除。

### 3.4 30% 压缩统计

- 展示原始正文总字数、整合后保留字数、压缩比。
- 压缩目标为 `整合后保留内容总字数 / 原始正文总字数 <= 30%`。
- 如果统计口径是粗略估算，必须在报告中说明。
- 压缩不能破坏教学完整性，需要保留关键前置依赖链路。

### 3.5 RAG 引用问答

- 用户输入医学问题后，系统检索教材片段并生成回答。
- 回答必须带教材、章节、页码、相关度和原文片段。
- 找不到依据时返回“当前知识库中未找到相关信息”。
- 模型或 embedding 不稳定时允许关键词检索兜底。

### 3.6 模拟教师反馈

- 明确标注反馈来源为“模拟教师反馈”。
- 至少支持保留、拆分、合并、删除中的一种修改。
- 反馈后能改变一条整合决策，并刷新图谱或决策列表。

## 4. 非功能需求

- 可复现：README 写明本地运行方式和环境依赖。
- 可部署：提交公网链接，不能使用 localhost。
- 安全：教材 PDF、`.env`、token、cookie、私钥不进入 GitHub。
- 可解释：核心决策必须可追溯到教材来源和规则说明。
- 稳定：API 失败或模型失败时前端不能白屏。

## 5. MVP 范围

MVP 优先使用 2 本教材跑通闭环：教材解析、章节结构展示、知识点抽取、单本图谱、跨教材整合决策、压缩比统计、带引用 RAG 问答和模拟教师反馈。

7 本教材完整处理作为报告和加分扩展，不作为第一阶段阻塞条件。

## 6. 明确不做

- 不承诺实时完整处理全部 824MB 教材。
- 不做复杂多 Agent 协作界面，只在架构文档中说明智能体分工。
- 不做 DOCX/Excel 解析，除非核心闭环已稳定。
- 不做复杂多视图图谱、桑基图或矩阵热力图。
- 不把模拟反馈描述为真实教师反馈。

## 7. 验收标准

- 本地可以打开前端，后端 `/health` 正常。
- 至少 2 本教材形成“解析 -> 图谱 -> 整合 -> 压缩统计 -> 引用问答”闭环。
- 图谱节点可查看来源和详情。
- RAG 回答包含教材、章节、页码引用。
- 压缩统计展示公式与结果。
- 文档能解释知识点粒度、重复判定、30% 压缩口径、教学完整性和反馈机制。
